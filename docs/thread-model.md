# Thread Model and Emulation Synchronization

This document explains the threading architecture of rNES and describes how the
emulator is kept in sync with real time. The information is derived from the Doxygen
documentation generated by `make docs` and the source of `src/interface/posix/rNES.c`.

---

## Overview

The reference POSIX/SDL implementation uses two POSIX threads and one semaphore:

| Symbol | Type | Purpose |
|--------|------|---------|
| `th_gui` | `pthread_t` | GUI thread — display, events, audio device |
| `th_emu` | `pthread_t` | Emulator thread — CPU/PPU/APU clocking |
| `wait_gui` | `sem_t` | Signals that `gui_init()` has completed |
| `quit_emulation` | `int` | Flag that tells the emulator thread to exit |

---

## Startup Sequence

The entry point (`main()` in `rNES.c`) initializes all emulator modules, then
creates the two threads. The semaphore ensures the emulator thread never starts
before the GUI (and audio device) is fully ready.

```
main()
  │
  ├── sbus_init()
  ├── sbus_set_cartridge(cartridge)
  ├── cpu_init()
  ├── ppu_init()
  ├── apu_init()
  ├── cpu_register_kill_cb(cpu_kill_cb)
  ├── controller_reset()
  ├── ppu_set_cartridge(cartridge)
  ├── cpu_reset()
  ├── ppu_reset()
  ├── apu_reset()
  │
  ├── pthread_create(th_gui)  ──► gui_th_loop()
  │       │
  │       ├── gui_init(scale)         ← opens window + audio device
  │       ├── sem_post(&wait_gui)     ← unblocks main thread
  │       └── gui_main_loop()         ← blocks here
  │
  ├── sem_wait(&wait_gui)     ← wait for GUI to be ready
  │
  ├── pthread_create(th_emu)  ──► emu_th_loop()
  │       │
  │       └── while (!quit_emulation)
  │               sbus_clock()        ← tight loop, no sleep
  │
  └── pthread_join(th_gui)    ← main thread waits here
```

## The Emulator Thread

```c
/** Emulation thread (main routine) */
static void *emu_th_loop(void *args)
{
    /* Emulation will be synchronized by the audio callbacks */
    while (!quit_emulation)
        sbus_clock();
    return NULL;
}
```

`sbus_clock()` is the heart of the emulation. It handles clocks from other modules and DMA transactions, specifically:

- Dispatches one tick to `cpu_clock()` — advances one CPU clock tick
- Dispatches one tick to `ppu_clock()` — calls `gui_set_pixel()` for each visible pixel.
- Dispatches one tick to `apu_clock()` — advances the APU; pushes PCM samples into the ring buffer when a new sample is ready.
- Handles OAM DMA transfers that may suspend the CPU (`cpu_suspend()` / `cpu_wakeup()`).

There is **no sleep or yield** in the emulator loop. The thread runs as fast as the
host CPU allows, relying solely on audio backpressure for real-time pacing.

## The GUI Thread

```c
/** GUI emulation thread (main routine)
 *  @param args  Pointer to an integer variable with scale factor */
static void *gui_th_loop(void *args)
{
    int *scale = (int *)args;
    if (gui_init(*scale) < 0) {
        log_err("Could not initialize GUI!\n");
        return NULL;
    }
    sem_post(&wait_gui);
    gui_main_loop();
    quit_emulation = 1;
    return NULL;
}
```

`gui_main_loop()` is responsible for:

1. Polling `SDL_PollEvent` and updating the joypad bitmasks used by `gui_read_joypad()`.
2. Uploading the pixel array (written by `gui_set_pixel`) into the SDL texture and calling `SDL_RenderPresent` once per frame.
3. Returning when the user closes the window or otherwise requests a quit.

When `gui_main_loop()` returns, the thread sets `quit_emulation = 1`. The emulator
thread checks this flag at the top of its loop and exits cleanly.

## Timing: Audio-Driven Synchronization

The NES master clock runs at ~21.477 MHz (NTSC). `sbus_clock()` is called once per
master clock cycle, meaning the emulator loop can execute tens of millions of
iterations per second on modern hardware — far faster than real time. Timing is
governed by the audio ring buffer.

### Data Flow

```
Emulator thread                     Audio callback (SDL audio thread)
───────────────                     ─────────────────────────────────
sbus_clock()
  └─► apu_clock()
        └─► rbuff_put(sample)  ──► rbuff_get(sample) ──► apu_get_samples()
                                        ↑                       │
                                   (blocks when empty)          ▼
                                                         audio_stream_put()
                                                                │
                                                                ▼
                                                          SDL audio device
                                                                │
                                                                ▼
                                                            Speaker
```

- The APU calls `rbuff_put()` to push a `int16_t` sample into the `ringbuffer_t`.
- The ring buffer (`ringbuffer_t`) is a FIFO; `rbuff_put()` blocks when
  the buffer is full (capacity = `AUDIO_BUFFER_LEN` = 2048 samples).
- The SDL audio callback fires at `AUDIO_SAMPLE_RATE` (44100) Hz and calls
  `apu_get_samples()`, which drains samples from the ring buffer via `rbuff_get()`
  and passes them to the speaker via `audio_stream_put()`.
- When the buffer is full, `rbuff_put()` stalls the emulator thread until the
  audio callback drains enough samples — naturally throttling emulation to real speed.

### Audio Callback Registration

During `apu_init()`, the APU registers `apu_get_samples` with the HAL via
`audio_set_cb(apu_get_samples, ...)`. The SDL audio device then invokes this
callback at ~21.5 times per second (44100 Hz ÷ 2048 samples/buffer).

### Why 44100 Hz?

The APU resamples from the NES master clock rate to `AUDIO_SAMPLE_RATE` = 44100 Hz,
a standard CD-quality rate supported by virtually all audio hardware. This value is
defined as a constant in `hal.h`.

## Porting Considerations

### Platforms Without pthreads

Replace `pthread_create` / `pthread_join` with your platform's task primitive:

| Platform | Task creation | Synchronization |
|---|---|---|
| FreeRTOS | `xTaskCreate` / `vTaskDelete` | `xSemaphoreCreateBinary` |
| Zephyr RTOS | `k_thread_create` | `k_sem` |
| Windows | `CreateThread` / `WaitForSingleObject` | `CreateSemaphore` |

### Platforms With a Push-Based Audio API

If the platform provides a write-style audio API rather than a callback-style one,
create a third thread that:
1. Calls `rbuff_get()` until `AUDIO_BUFFER_LEN` samples are collected.
2. Calls `audio_stream_put()` to write them to the device.
3. Repeats, simulating the 44100 Hz callback cadence.

### Reducing CPU Usage on Low-Power Targets

Because the emulator loop is a busy spin, it can consume 100% of one CPU core. On
targets where this causes thermal or power issues, insert a short `usleep` or
`vTaskDelay` inside `emu_th_loop` when the ring buffer is nearly full. This must be
tuned carefully to avoid audio underruns — even a few milliseconds of stall can cause
an audible glitch.
